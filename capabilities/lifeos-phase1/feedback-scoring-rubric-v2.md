# Hermes 记录入库反馈评分规约 V2

日期：2026-04-12  
状态：Active  
来源：hiddenwangcc 最新人工判断标准

## 1. 这份规约解决什么

这份规约用于统一两件事：

1. Codex / Hermes 以后如何给一条记录入库反馈打分
2. Hermes 生成反馈时，什么叫“真正贴近 hiddenwangcc”

它不是通用的文案评价标准，而是：

`针对 hiddenwangcc 的反馈质量标准`

也就是说，后续任何一条入库反馈，都要尽量符合这里定义的判断方式。

## 2. 总原则

### 2.1 标题不是摘要，而是“场景中的定位结论”

主标题不能只是：

- 可复用的方法
- 很重要的一条内容
- 一个值得沉淀的判断

这种抽象句子。

因为它们虽然正确，但：

- 1 个月后人类几乎记不住
- 无法帮 hiddenwangcc 快速回到当时的具体背景
- 无法形成“这是哪一类人生运行片段”的索引

主标题必须回答：

- 这条内容是在什么背景下形成的
- 用户当时到底在做什么
- 当前处在什么阶段
- 用户对结果持什么预期

也就是：

`背景 + 动作 + 阶段 / 判断`

例如：

- 连续调试卡片反馈功能，当前初期阶段预期不会很高

### 2.2 副标题必须是一个问题

副标题不是解释句，也不是标签串。

副标题必须满足：

- 是一个明确问题
- 主标题是这个问题的答案

例如：

- 主标题：连续调试卡片反馈功能，当前初期阶段预期不会很高
- 副标题：当前卡片反馈功能调试处于什么阶段？

### 2.3 反馈必须围绕“内容与你之间的关系”

很多反馈的错误，不是信息错，而是关系错。

错误方式：

- 只说这条内容是什么
- 只说它很重要
- 只说它值得沉淀

正确方式：

- 这条内容为什么会被你说出来
- 这条内容和你当前正在推进的系统之间是什么关系
- 这条内容如何体现你的判断、预期、姿态、方法、张力

## 3. 评分维度

每条记录反馈，按 6 个维度评分。

每项 0 到 10 分，总分取平均。

### 3.1 标题记忆度

看点：

- 1 个月后再看，能不能立刻想起当时在发生什么
- 是否包含“背景 + 动作 + 阶段 / 判断”
- 是否避免过度抽象

低分特征：

- “一条可复用的方法”
- “很重要的内容”
- “值得沉淀的理解”

高分特征：

- 能还原场景
- 能直接成为人类记忆索引

### 3.2 问答配对度

看点：

- 副标题是否真的是问题
- 主标题是否真的是这个问题的答案

低分特征：

- 副标题不是问题
- 主副标题之间没有问答关系

高分特征：

- 一眼就能看出“这是在回答哪个问题”

### 3.3 内容贴身度

看点：

- 反馈是否真正围绕 raw 内容展开
- 有没有脱离具体内容，退化成通用句

低分特征：

- 能套在很多条内容上
- 换一条记录也几乎不用改

高分特征：

- 只能属于这一条
- 能看出对具体语境的理解

### 3.4 人我关系度

看点：

- 是否看到“这条内容和 hiddenwangcc 之间的关系”
- 是否看到用户的判断方式、推进姿态、真实预期、系统位置

低分特征：

- 只解释内容，不解释人与内容的关系

高分特征：

- 看见用户为什么会在这个时刻说这句话
- 看见这条内容和用户当前系统建设之间的关系

### 3.5 结构解释度

看点：

- 表达类型是否长期可积累
- 情绪是否给出真实词汇
- 能量是否使用大卫霍金斯意识能量层级
- 当前状态与需要是否具体

要求：

- 表达类型：持续沉淀和演化
- 情绪判断：必须出现真实情绪词
- 当前能量：使用霍金斯层级名称或对应层级表达
- 状态 / 需要：保持具体

### 3.6 行动开发感

看点：

- “关键内容 / 真正重要的地方 / 下一步建议 / 继续帮什么”是否有明显的系统开发感
- 是否像真正处在 LifeOS / Hermes 构建过程中的反馈

低分特征：

- 像开发前灵感记录
- 像泛泛建议
- 缺少工程推进感

高分特征：

- 能看出下一步是设计、调试、验收、沉淀、资产化还是模块接入
- 建议与系统开发阶段匹配

## 4. 当前这条样例的打分方式

样例 raw：

`昨天晚上抱着 codex 睡的，早晨起来说卡片的入库功能差不多好了，我想试试看。我觉得应该不会那么理想，毕竟有大量的信息的维度还没有调试，AI 不可能弄得那么清楚的。尝试一次。`

现有评分判断：

- 标题记忆度：低
- 问答配对度：低
- 内容贴身度：中
- 人我关系度：低
- 结构解释度：中
- 行动开发感：低

## 5. 对 Hermes 的生成要求

### 5.1 标题生成公式

优先从 raw 中提取：

1. 当前正在做什么
2. 处在什么阶段
3. 对结果持什么预期
4. 这件事属于哪个系统动作

推荐公式：

- `动作 + 当前阶段 + 预期 / 判断`
- `连续动作 + 当前状态 + 风险判断`

### 5.2 副标题生成公式

从主标题反推问题。

推荐公式：

- 当前 X 处于什么阶段？
- 这次 Y 的真实判断是什么？
- 用户此刻在推进的核心动作是什么？

### 5.3 “我对这条的判断”生成要求

- 表达类型：保留可积累性
- 情绪判断：必须是真实词汇，不可为空泛
- 当前能量：用霍金斯意识能量层级
- 当前状态：描述用户所处的运行状态
- 你此刻更需要的：必须具体

### 5.4 “关键内容”生成要求

不能说机器话。

禁止：

- 这不只是记录一个事实
- 这条内容很重要
- 值得沉淀

应当输出：

- 用户此刻真正形成的判断
- 用户对当前阶段的真实预期
- 用户在推进系统时的姿态与方法

### 5.5 “真正重要的地方”生成要求

必须回答：

- 为什么这条内容对你重要
- 这条内容和你当前正在建设的 LifeOS / Hermes 有什么关系

### 5.6 “下一步行动方向”生成要求

优先进入开发语境：

- 调试
- 验收
- 分层
- 补字段
- 调提示
- 调 event schema
- 优化路由
- 增加卡片区块
- 形成资产化留存

不能停留在泛泛的灵感整理上。

## 6. 下一次打分时的执行规则

Codex 以后给入库反馈打分时，必须：

1. 先按 6 维评分
2. 再说总分
3. 明确指出：
- 最强项
- 最弱项
- 是否贴近 hiddenwangcc
- 下一步优先优化哪一块

## 7. 当前阶段的改进方向

基于这次人工判断，下一阶段优先级如下：

1. 标题系统重写
2. 问答配对生成
3. 情绪词与霍金斯能量层级接入
4. 把“关键内容 / 重要性 / 下一步建议”改成更强的开发语境
5. 让反馈持续围绕“内容与你之间的关系”展开
