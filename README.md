## 1. 这个文件夹是什么

这是 `hiddenwangcc` 的专属 Agent 自建能力目录。

它的目标不是单纯放文档，而是同时承载：

- 开发过程的讨论留存
- 能力设计文档
- 流程与规范
- 脚本与辅助工具索引
- 能力发布与版本记录
- 给新 Agent 的接管入口

一句话说：

`这里是“你自己建设 AI 代理人能力”的可迁移资产库。`

## 2. 这里的原则

### 2.1 先留痕，再实现

每一次关键讨论、设计判断、重要来回对聊，都要沉淀成文档。

### 2.2 文档和能力必须同源

一个能力如果已经开始开发，就应该在这里有：

- 讨论记录
- 能力说明
- 当前状态
- 相关脚本或脚本索引

### 2.3 可迁移优先

这里的内容必须让未来的新 Agent 能重新读取、重新理解、重新接手。

### 2.4 Git 版本管理优先

这个目录里的变更必须通过 git 留下历史，并最终推到 GitHub。

## 3. 目录结构

- `AGENT-ONBOARDING.md`
  - 新 Agent 第一次接手时优先阅读的入口
- `REGISTRY.yaml`
  - 当前能力、流程、记录的总索引
- `sessions/`
  - 每一次关键开发讨论与推进记录
- `capabilities/`
  - 每个能力的说明、状态、相关实现路径
- `workflows/`
  - 规范化流程，例如版本控制、GitHub 发布、会话留痕规则
- `templates/`
  - 新增记录时使用的模板
- `scripts/`
  - 这里优先放以后新增的专属脚本

## 4. 当前执行规则

从现在开始，凡是以下内容，都优先在这里留存：

- 你和 Agent 关于“能力开发”的关键讨论
- 新能力的需求与设计
- 重要结构决定
- 实际开发产物的说明与索引
- 发布与版本变化

## 5. 与 GitHub 的关系

目标远端仓库名称：

`AI 代理人自建功能`

推荐 GitHub repo slug：

`runwithcc/ai-agent-custom-capabilities`

当前现实情况：

- 本地仓库已存在
- 但当前工作区还没有配置远端
- 也还没有接到该 GitHub repo

所以当前正确路径是：

1. 先把本地标准结构建好
2. 让后续新增资产都进入这个目录
3. 再把本地仓库绑定到目标 GitHub repo
4. 后续每次能力更新都走 git 提交与 push

## 6. 当前最重要的一句话

未来即便更换 Hermes、OpenClaw、Codex，甚至更换整个主程序，只要这个文件夹和 git 历史还在，你的 Agent 自建能力资产就还在。
